# Домашнее задание #10: Knex Query Builder + Prisma ORM

## Вопросы для самопроверки (в README.md)

1. Почему сервисы и контроллеры **не изменились** при замене stub → Knex → Prisma? Какой принцип это обеспечивает?

    В трёхслойной архитектуре верхние слои могут знать о нижних, нижние слои не должны знать о верхних. Помимо этого все слои написаны заранее так, чтобы связанность между слоями была минимальна.
2. Что такое N+1 проблема? Покажите «плохой» и «хороший» вариант из вашего кода.

    Проблема N+1 возникает, когда приложение выполняет 1 запрос для получения списка объектов, а затем по одному дополнительному запросу для каждого объекта из этого списка.
    Плохо:

    ```
    const posts = await prisma.post.findMany();

    for (const post of posts) {
    post.author = await prisma.user.findUnique({
        where: { id: post.userId },
    });
    }
    ```
    Лучше:

    ```
    const posts = await prisma.post.findMany({
        include: {
            author: {select: { id: true, name: true }},
        },
    });
    ```

3. Когда вы выберете Knex, а когда Prisma? (3–4 предложения с примерами)

    Knex - нужны сложные запросы (например BI-платформа, система выставления счетов, сложный поисковой сервис).
    Prisma - важна скорость и удобство разработки (например трэкер задач, CRM, интернет магазин).
4. `prisma migrate dev` vs `prisma migrate deploy` — в чём разница?

    `prisma migrate dev` - создать миграцию + применить миграцию.
    `prisma migrate deploy` - просто применить миграцию.

5. Как Knex защищает от SQL-инъекций?

    С помощью параметризованных запросов. 

    ```
    const user = await db("users")
    .where({ email }); // сгенирирует SELECT * FROM users WHERE email = ?
    ```

    Значение переменной email будет переданно в запрос как строка.
